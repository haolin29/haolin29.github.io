NYU Big Data Final Review

Info for final exam:

The final exam focuses on post-midterm material mostly, but familiarity with pre-midterm slides and readings will be useful

You do not need to re-read research papers, but you should re-read the book sections that were assigned since the midterm exam (Except the Facebook case study - it is not on the exam)

You should review post-midterm slides

Review the Hive and Impala homework - re-run the commands

An understanding of the roles of the mapper and reducer is necessary.

Review your old MapReduce programs, re-run them, read the early chapters with the weather example

Familiarity with the Wordcount program will be helpful

There will likely be at least one program to write and/or interpret

Understand MapReduce 1 architecture - know the major compute components: JobTracker, TaskTrackers

Understand YARN/MapReduce 2 architecture - know the major compute components: ResourceManager, NodeManager, Application Master

Understand how Hadoop (HDFS + MapReduce) works

  • The client, which submits the MapReduce job.
  • The YARN resource manager, which coordinates the allocation of compute resources on the cluster.
  • The YARN node managers,which launch and monitor the compute containers on machines in the cluster.
  • The MapReduce application master, which coordinates the tasks running the Map‐ Reduce job. The application master and the MapReduce tasks run in containers that are scheduled by the resource manager and managed by the node managers.

Understand the 4 behaviors of Autonomic Computing

Self-configuring: all configuration should happen with little or no intervention by the user

Self-healing: When a progress, service or node stop working, it should repair itself.

Self-protecting: Continually assess its own health. Redundant, cooperating health checkers can health check each other.

Self-optimizing: Continually assess its own performance.

perform load-balanceing operation

request resources to maintain a desired level of performance

There are no HBase, Flume, or Zookeeper programs to write, but you should study the slides

Flume is designed for high-volume ingestion into Hadoop of event-based data.

There are no Twitter API questions

Review the Spark homework and slides

Spark is a cluster computing system for fast-running analytics.

Unlike map reduce, spark efficiently handle multi-pass applications.

RDD = resilient distributed dataset

  • hold the data that being processed
  • RDDs can store in memory and can persist from query to query

RDDs is not stored with redundancy, but recovery from failure is still possible

  • fault tolerance is provided using the linage information tracked by RDD

  • RDD can re-build itself using linage information

    $ file1 = sc.textFile(“myfile.txt”)
    $ westLine = file1.filter(lambad line: “West” in line).count()

RDD can persist

  • using the persist() or cache() method
  • to disk
  • by replicating it across multiple nodes

All transformation in spark is lazy transformations.

  • the result is not immediately computed
  • when a result need to be returned to the driver program, all transformation are computed - an action trigger the transformation

what is Oozie and how does help

Oozie: workflow in hadoop can be scheduled using oozie, using for hadoop and spark workflow control

oozie is useful in organizing the processing stage.

  • launch processing either in parallel or sequentially
  • prevent processing step from running until earlier prerequisite steps have completed
  • allow map reduce, pig, spark hive etc processing step to be interleaved and run automatically as scripts
  • Oozie workflow is captured in an XML file.

Understand how failures are detected/handled

MapReduce 1:

  • task failure
    • jvm reports, task tracker handle failure
    • task tracker notices the lack of update, kill jvm process
    • Job tracker notified through task tracker heartbeat message
  • tasktracker failure
    • fails to send heartbeats regularly, job tracker reschedules completed tasks
    • a job tracker can blacklist the task tracker if more than four tasks from some job fail
  • job tracker failure: SPOF in MR1

YARN:

  • task failure
    • task’s JVM reports error back to parent AM, AM marks as failed and freeing up resources
    • JVM exit unexpectedly, NodeManager notices, AM attempt as failed
    • AM notices lack of update, NM marks as failed
    • AM reschedules execution
  • nodemanager failure
    • how to detect? fails to send heartbeat regularly, RM will detect
    • how to handle? RM remove failed NM from the pool
    • if the number of failures for the application is high, an AM can blacklist the NM
  • application master failure
    • RM detect the failure from heartbeat
    • the new AM will use job history to recover state of tasks
  • resource manager failure
    • the most serious failure

Understand JVM reuse in MR1 vs. uber tasks in YARN

Understand how map/reduce task progress is measured

Understand Hive, Zookeeper, MapReduce concepts

Understand advantages/disadvantages of running a Hadoop cluster in a public cloud

high availability

HDFS high availability only works if replicas are wisely distributed: but we don’t know the network topology

Performance

MapReduce map tasks perform best when co-located with the HDFS block to be processed, or as near as possible to the block if co-location is not an option

Only if Hadoop co-locates the map task on the same VM with its data block do we know with certainty that the map task is optimally placed in a public cloud

If the map task cannot be co-located with the data, there is no way to know whether the VM on which the map task will run is in the same server, same rack, or same datacenter as the data block

Sqoop and Oozie material: Be able to describe the purpose of these tools.

Sqoop purpose: use sqoop to move data between HDFS and external database, such as mysql

$ sqoop import --connect jdbc:mysql://localhost/my_db_name --table my_table

NoSQL

什么是NoSQL？

  • Non-relational databases
  • do not support sql
  • do not guarantee acid properties
  • designed to be distributed across many nodes
  • provide eventual consistency
  • easier to scale up than with relational databases

NoSQL database

  • developed to manage large volumes of data that do not necessarily follow a fixed schema
  • data is fragmented and stored across many servers
  • satisfy realtime constraints is the priority
  • joins are not needed, recommendation is that data be de-normalized

HBase

  • a column family is a grouping of columns
  • columns that belong to a given columns family are physically stored together
  • values are stored in HBase cells
  • cells can be versioned: row key + columnFamily:columnName + Timestamp
  • region: A subset of an HBase table’s rows starting with a starting row key and exclusive of the ending row key
  • read are assisated by the block cache
  • write are assisated by the write ahead log, and then to memstore for quick lookup, when the amount of data in memstore reach a threshold, it is flushed to disk

Hive

  • data warehouse framework
  • built on top of map reduce, contrast with HBase, which is built directly on top of HDFS
  • user data is organized into tables
  • metastore: special table for storing metadata such as table schema
  • schema on read
  • a hive table is essentially an HDFS directory containing one or more files comprise the table
  • don’t support deletes and updates
  • parition: divided a table based on the value of a parition column

ZooKeeper takeaway

  • four behavior of autonomic computing
    • self-configuring
    • self-healing
    • self-protecting
    • self-optimizing
  • for support high availiabilty
  • why we need it?
    • to learn status of self and peers
    • to obtain reliable and timely information
    • simplified user experience
    • difficult to identify root cause of the failure
  • what is zk?
    • hadoop distributed coordination service
    • provide tools for simple and safe handling of partial failures
    • facilitate loose coupling

YARN

  • uberization mean that the tasks are too small to run in parallel, the AM will run the tasks serially in its own JVM

什么是React？

很多人可能会对这个问题感到困惑。React 其实不是Augular.js 或者 Backbone.js 那样的MVC 框架。 它只是MVC中的V — 也就是表现层， react 只做一件事，这也是符合UNIX的思路。按官网描述，其出发点为：

用于开发数据不断变化的大型应用程序（Building large applications with data that changes over time）。

React最初来自Facebook内部的广告系统项目，项目实施过程中前端开发遇到了巨大挑战，代码变得越来越臃肿且混乱不堪，难以维护。于是痛定思痛，他们决定抛开很多所谓的“最佳实践”，重新思考前端界面的构建方式，于是就有了React。

tmux cheatsheet

新建会话

tmux new -s session-name

离开会话

ctrl-a d

tumx detach

接入之前的会话

tmux a

tumx a -t session-name

关闭会话

tumx kill-session -t session-name

tumx killall 同时关闭tmux

配置文件

~/.tmxu.conf

窗口操作:

c 创建新窗口
& 关闭当前窗口
数字键 切换到指定窗口
p 切换至上一窗口
n 切换至下一窗口
l 前后窗口间互相切换
w 通过窗口列表切换窗口
, 重命名当前窗口，便于识别
. 修改当前窗口编号，相当于重新排序
f 在所有窗口中查找关键词，便于窗口多了切换

面板操作:

“ 将当前面板上下分屏
% 将当前面板左右分屏
x 关闭当前分屏
! 将当前面板置于新窗口,即新建一个窗口,其中仅包含当前面板
Ctrl+方向键 以1个单元格为单位移动边缘以调整当前面板大小
Alt+方向键 以5个单元格为单位移动边缘以调整当前面板大小
空格键 可以在默认面板布局中切换，试试就知道了
q 显示面板编号
o 选择当前窗口中下一个面板
方向键 移动光标选择对应面板
{ 向前置换当前面板
} 向后置换当前面板
Alt+o 逆时针旋转当前窗口的面板
Ctrl+o 顺时针旋转当前窗口的面板
z tmux 1.8新特性，最大化当前所在面板

为什么开始写博客

Posted on 2016-02-28   |   In 想法   |  

受一篇文章启发，决定开始写博客，把自己的想法记录下来，形成知识脉络。
文章转载如下：

为什么你应该（从现在开始就）写博客

（一）为什么你应该（从现在开始就）写博客

用一句话来说就是，写一个博客有很多好处，却没有任何明显的坏处。

让我说得更明确一点：用博客的形式来记录下你有价值的思考，会带来很多好处，却没有任何明显的坏处。Note：碎碎念不算思考、心情琐记不算思考、唠唠叨叨也不算思考、没话找话也不算思考，请以此类推。

下面是我个人认为写一个长期的 价值博客 的最大的几点好处：

1) 能够交到很多志同道合的朋友。我自己既写博客，也读别人的博客，在这个时代，对于生活中的绝大多数人来说，拓宽朋友圈子的途径几乎只有一个，通过网络，而如何在网络中寻找到气味相投的朋友，如何判断别人和自己是否有共同语言？显然，通过天天在SNS上碎碎念的那些日记是难以做到的。我佩服的一些朋友几乎全都是长期用博客记录想法的人，因此，和他们即便不打照面，也是心照不宣。即便素未谋面也能坐下来就聊得热火朝天。

为什么博客在结交志同道合的朋友方面的潜力要远胜于原始的交谈方式？很简单，第一，博客无地域限制，整个互联网上从A到B只有一个点击的距离，而传统的建立朋友圈子的方法则受到地域限制。第二，也是更重要的一点，即如果按照以前结交朋友的方式，需要互相聊天，交流观点，然后才逐渐熟悉起来，这需要一个较长的过程，而且更糟糕的是，当你遇到另一个陌生人，又要把整个过程重复一次，表达你已经对老友表达过的那番想法。可博客却做到了“一次表达，无数次阅读”，当我看到一个写了好几年的博客，看完了之后我仿佛和这个人交谈了很久，用程序员们喜欢听的话来说就是，“博客极大地增强了话语的复用性”。

我曾在CSDN上写了近六年的博客，在一年半前建立了一个Google Groups（TopLanguage），由于我的博客的长期阅读者都是互相有共同语言的，因此这个Group一开始就热火朝天，而高质量的技术讨论则进一步吸引了更多的牛人的参与，雪球滚起来之后，就很难停下来了，将近一年半下来，从这个Group的讨论中我获益良多[1]。而对于非程序员朋友，科学松鼠会则是一个很好的例子。
